optimising productivity 18/11/2010
What determines productivity, and how is it measured?
What tools are there to increase productivity and reduce development costs?
Will code coverage, complexity and dependency management save time and money?
Each company will determine productivity in a different way, income per developer, dare we say lines of code, and for internal development this gets rather more difficult. So productivity really needs to be determined with the backdrop of the environment in view. Preferably by knowledgeable persons that can understand the relationships between code metrics and financial dynamics of products.
The costs associated with development are 20% for initial development and 80% for maintenance over a 5 year period, assuming that the average application has a life span of 5 years. If development is accelerated by 20% and we assume that there is a direct and proportional relationship between quality and development speed(which there is of course) and we assume an increase of 2% maintenance costs for every 1% drop in code quality then we do a little calculation:

Initial development cost : 100 000
Maintenance costs : 400 000
Code quality : 100%

Total cost : 500 000

Now we change initial development time taken and the code quality.

Initial development cost : 100 000 * 80%
Maintenance costs : 400 000 * 140%
Code quality : 80%

Total cost now : (80 000 + 560 000) 640 000

We can see an immediate and not trivial change in the final cost of the project. The initial development seems very productive, and from a managerial perspective the development team will be praised for their results, bravo, but in a couple of years not only will it become more difficult to find developers that will toilet clean the code(and I dare say more expensive to employ such developers) but the project will seriously be in danger of running at a loss.
How can project leaders, managers, and developers check their code quality? It is a requirement that there is a unit test environment for the project. Code coverage in unit tests should be over 80%. Code metrics can be checked and verified that they fall into acceptable levels, like dependency and complexity.
What tools are available to measure complexity, and code quality? Well there are code coverage, complexity and dependency tools like Serenity. Dependency tools like JDepend, and code coverage tools like Cobertura. There are static code analysis tools like FindBugs, CheckStyle and Crap4j. Like any tool each needs to be studied to understand the statistics that are generated by the tools. And of course this is exactly what your job is if you are a manager of a project. Clearly. However to facilitate easy inspection as always the continuous integration(CI) server is a good idea. Servers for CI are numerous and generally free or at least very cost effective. Hudson is a fine example. The continuous integration server is an area on the intranet where managers and developers can see immediately what the code metrics are and where to improve. They can evaluate each others code serving as code review as well.
Looking around at the projects that are profitable and have flourished and looking at the source you will notice that there are things in common. Indeed, the unit test code coverage is above 50%, the JavaDoc is complete and well done and the complexity is below 10.
Just some examples of projects that have stood the test of time are ASM Byte Code Library(have a look at the ClassVisitor for example), Eclipse, Lucene, Hudson, FindBugs, the list goes on and on. Conversely one only need to look at the same metrics of failed projects and one will see no unit test coverage, no JavaDoc and complexity over 20.
Clearly the choice is a simple one. Don't rush the initial development, you will pay for it dearly in the long run. Every language has some expression for this the English one being 'A stitch in time saves nine', Dutch 'Haast en spoed zijn selde goed'.
Prior to the famous GOF pattern book, patterns didn't exist in code at all. No. Of course they did, everywhere. But they weren't so nicely packaged as in Patterns. Critical to any product is the architecture. And by architecture, not the UML diagrams,and not the push/pull discussions in the meeting room, no, architecture is the patterns employed. Although it is hard to say exactly what the empirical benefit and ROI is of applying and employing patterns in software, and there are scant research efforts into this as yet, every experienced programmer knows the difference between trying to reverse engineer a piece of code that was written in the 5 minutes before delivery, 50% hard coded and written in a panic attack; to a piece of code where the business logic maps directly to the code pattern, well documented and re- factored to an absolute simplicity, unit tests for every major area of the code and 80% code coverage.
How much easier, faster, nicer will that patterned code be? Could it be 20% nicer? Hard to say, but I am sure it will be nicer. Patterns are not restricted to the 'architects', no, senior programmers must know at least 20 patterns, and have implemented at least 10 before they can be called senior. Patterns go back right to the beginning, in fact throughout the Patterns book they refer to the first known account of such and such pattern. This is a great and mandatory book for any programmer. Of course everyone has their own opinion, in mine, the cost of a long term project can be reduced by 40% if the code is well written, JavaDoced, unit tested, patterned, continuously integrated and re-factored regularly. Naturally a book on re-factoring wouldn't go wanting either.

optimising productivity 18/11/2010
What determines productivity, and how is it measured?
What tools are there to increase productivity and reduce development costs?
Will code coverage, complexity and dependency management save time and money?
Each company will determine productivity in a different way, income per developer, dare we say lines of code, and for internal development this gets rather more difficult. So productivity really needs to be determined with the backdrop of the environment in view. Preferably by knowledgeable persons that can understand the relationships between code metrics and financial dynamics of products.
The costs associated with development are 20% for initial development and 80% for maintenance over a 5 year period, assuming that the average application has a life span of 5 years. If development is accelerated by 20% and we assume that there is a direct and proportional relationship between quality and development speed(which there is of course) and we assume an increase of 2% maintenance costs for every 1% drop in code quality then we do a little calculation:

Initial development cost : 100 000
Maintenance costs : 400 000
Code quality : 100%

Total cost : 500 000

Now we change initial development time taken and the code quality.

Initial development cost : 100 000 * 80%
Maintenance costs : 400 000 * 140%
Code quality : 80%

Total cost now : (80 000 + 560 000) 640 000

We can see an immediate and not trivial change in the final cost of the project. The initial development seems very productive, and from a managerial perspective the development team will be praised for their results, bravo, but in a couple of years not only will it become more difficult to find developers that will toilet clean the code(and I dare say more expensive to employ such developers) but the project will seriously be in danger of running at a loss.
How can project leaders, managers, and developers check their code quality? It is a requirement that there is a unit test environment for the project. Code coverage in unit tests should be over 80%. Code metrics can be checked and verified that they fall into acceptable levels, like dependency and complexity.
What tools are available to measure complexity, and code quality? Well there are code coverage, complexity and dependency tools like Serenity. Dependency tools like JDepend, and code coverage tools like Cobertura. There are static code analysis tools like FindBugs, CheckStyle and Crap4j. Like any tool each needs to be studied to understand the statistics that are generated by the tools. And of course this is exactly what your job is if you are a manager of a project. Clearly. However to facilitate easy inspection as always the continuous integration(CI) server is a good idea. Servers for CI are numerous and generally free or at least very cost effective. Hudson is a fine example. The continuous integration server is an area on the intranet where managers and developers can see immediately what the code metrics are and where to improve. They can evaluate each others code serving as code review as well.
Looking around at the projects that are profitable and have flourished and looking at the source you will notice that there are things in common. Indeed, the unit test code coverage is above 50%, the JavaDoc is complete and well done and the complexity is below 10.
Just some examples of projects that have stood the test of time are ASM Byte Code Library(have a look at the ClassVisitor for example), Eclipse, Lucene, Hudson, FindBugs, the list goes on and on. Conversely one only need to look at the same metrics of failed projects and one will see no unit test coverage, no JavaDoc and complexity over 20.
Clearly the choice is a simple one. Don't rush the initial development, you will pay for it dearly in the long run. Every language has some expression for this the English one being 'A stitch in time saves nine', Dutch 'Haast en spoed zijn selde goed'.
Prior to the famous GOF pattern book, patterns didn't exist in code at all. No. Of course they did, everywhere. But they weren't so nicely packaged as in Patterns. Critical to any product is the architecture. And by architecture, not the UML diagrams,and not the push/pull discussions in the meeting room, no, architecture is the patterns employed. Although it is hard to say exactly what the empirical benefit and ROI is of applying and employing patterns in software, and there are scant research efforts into this as yet, every experienced programmer knows the difference between trying to reverse engineer a piece of code that was written in the 5 minutes before delivery, 50% hard coded and written in a panic attack; to a piece of code where the business logic maps directly to the code pattern, well documented and re- factored to an absolute simplicity, unit tests for every major area of the code and 80% code coverage.
How much easier, faster, nicer will that patterned code be? Could it be 20% nicer? Hard to say, but I am sure it will be nicer. Patterns are not restricted to the 'architects', no, senior programmers must know at least 20 patterns, and have implemented at least 10 before they can be called senior. Patterns go back right to the beginning, in fact throughout the Patterns book they refer to the first known account of such and such pattern. This is a great and mandatory book for any programmer. Of course everyone has their own opinion, in mine, the cost of a long term project can be reduced by 40% if the code is well written, JavaDoced, unit tested, patterned, continuously integrated and re-factored regularly. Naturally a book on re-factoring wouldn't go wanting either.
